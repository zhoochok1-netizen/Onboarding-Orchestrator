# Презентация проекта — Onboarding Orchestrator

## Структура выступления (~10–15 мин)

---

## 1. Проблема (1 мин)

> «Онбординг — это хаос. HR оформляет документы, IT выдаёт доступы, руководитель ставит задачи, наставник проводит встречи. Без единой системы задачи теряются, сроки нарушаются, новичок не понимает что происходит. Нужен оркестратор.»

**Ключевое:** показать что это не просто "ещё один таск-трекер", а координация между 5 типами участников.

--- 

## 2. Решение — что мы построили (2 мин)

> «Onboarding Orchestrator — full-stack платформа, которая:»

- **Координирует** всех участников через единую доску задач
- **Автоматизирует** создание маршрутов из шаблонов
- **Отслеживает** SLA в реальном времени (зелёный/жёлтый/красный)
- **Даёт новичку** AI-помощника, который знает его контекст
- **Навигирует** по базе знаний компании через RAG

**Демо-данные:** 22 сотрудника, 10 онбордингов, 100 задач, 15 документов — всё работает out of the box.

---

## 3. Демо — показ приложения (5–7 мин)

### 3.1. Вход — ProfileSelector
- Показать экран выбора профиля
- Объяснить 5 ролей: HR, Руководитель, IT, Наставник, Новичок
- «Каждая роль видит только своё — фильтрация на бэкенде»

### 3.2. HR-менеджер → Дашборд
- Статистика: онбординги, задачи, SLA
- SLA-бары (зелёный/жёлтый/красный)
- Карточки онбордингов → клик → детальная страница
- Кнопка **«+ Запустить онбординг»** → модалка с выбором шаблона и редактором задач
- Кнопка **«⚙️ Автопинг»** → настройка уведомлений

### 3.3. Доска задач
- Фильтры: по ответственному, статусу, SLA
- Клик по задаче → модалка: статус, SLA, дедлайн, участники, кнопка «Пинг»
- Смена статуса через кастомный dropdown
- «Все селекты кастомные — никаких нативных элементов»    

### 3.4. Шаблоны
- 4 готовых шаблона (разработчик, маркетолог, продажи, дизайн)
- Кнопка **«+ Создать шаблон»** → двухколоночная модалка (этапы слева, задачи справа)
- Редактирование и удаление существующих

### 3.5. AI Чатбот ⭐ (главная фича)
- Спросить: **«Что мне нужно сделать?»**
- Показать: ёмкий ответ с кликабельными ссылками
- Кликнуть на задачу → модалка с деталями
- Кликнуть на новичка → модалка онбординга
- Кликнуть на документ → модалка с содержимым
- Спросить про процессы: «Как получить доступы?», «ДМС»
- Показать источники из базы знаний

### 3.6. База знаний
- 15 документов (СОП, ПВТР, ЛНА, гайды)
- Поиск через RAG (TF-IDF)
- «Спросить AI» — генерация ответа с источниками
- HR может загружать новые документы

### 3.7. Переключиться на Новичка
- Показать что интерфейс упрощён
- Дашборд показывает только свой онбординг
- Доска задач — только свои задачи
- AI-чатбот знает контекст новичка

## 4. Архитектура (2 мин)

### Стек
| Слой | Технология |
|------|-----------|
| Frontend | React 19, React Router, React Markdown |
| Backend | Python 3.12, FastAPI, Pydantic |
| AI | OpenRouter → Gemini 2.5 Flash |
| RAG | Собственный TF-IDF pipeline |
| Deploy | Docker, docker-compose, nginx |

### Схема
```
React (SPA) → REST API (FastAPI) → Mock Data Layer
                    ↓
              RAG Pipeline (chunking → TF-IDF → top-K)
                    ↓
              AI Gateway (OpenRouter → Gemini)
```

### Ролевая фильтрация
- Фронтенд отправляет `X-User-Id` в каждом запросе
- Бэкенд фильтрует данные по роли: HR видит всё, менеджер — свой отдел, остальные — свои задачи
- Это не просто скрытие на фронте — данные реально не приходят

---

## 5. AI и RAG — как работает (2 мин)

### RAG Pipeline
1. **15 документов** разбиваются на **86 чанков** (по секциям/параграфам)
2. **TF-IDF индекс** — 792 токена, русскоязычная токенизация
3. При запросе — **скоринг** каждого чанка, берём top-5
4. Только релевантные чанки отправляются в LLM (экономия токенов)

> «Не отправляем всю базу знаний — только то, что нужно. Это и быстрее, и дешевле.»

### Контекст чатбота
- Каждый запрос к AI включает:
  - Данные пользователя (роль, отдел, задачи, онбординги)
  - ID задач и онбордингов (чтобы AI мог генерировать кликабельные ссылки)
  - Релевантные чанки из базы знаний (RAG)
- Системный промпт: краткие ответы, Markdown, обязательные ссылки, источники

### Кликабельные ссылки
- AI генерирует: `[Задача](task:id)`, `[Новичок](onboarding:id)`, `[Документ](knowledge:doc-id)`
- Фронтенд перехватывает эти ссылки и открывает модалки
- «Чатбот не просто текст — это интерактивный интерфейс»

## 6. Принятые решения (1 мин)

**Почему mock-данные, а не БД?**
> «1.5 дня. Выбор: потратить время на PostgreSQL + миграции + seed, или показать максимум функциональности. Выбрал второе. Архитектура готова к замене — нужно подменить data layer.»

**Почему собственный RAG, а не LangChain?**
> «Полный контроль, ноль лишних зависимостей. TF-IDF достаточен для 15 документов. При масштабировании — заменяем на FAISS/Chroma без изменения API.»

**Почему OpenRouter?**
> «Единый интерфейс к 100+ моделям. Переключение Gemini → Claude → GPT — одна строка в .env.»

## Возможные вопросы

**Q: Почему нет базы данных?**
> A: Осознанный выбор: mock-данные позволили сфокусироваться на UX и AI-фичах. Data layer абстрагирован — замена на PostgreSQL + SQLAlchemy не затронет ни один UI-компонент.

**Q: Как масштабировать RAG?**
> A: Заменить TF-IDF на FAISS или ChromaDB. API (`rag.retrieve()`, `rag.build_context()`) останется прежним. Для 1000+ документов добавить эмбеддинги через sentence-transformers.

**Q: Безопасность? X-User-Id в заголовке — это же не авторизация.**
> A: Верно, это демо-механизм. В проде заменяется на JWT + middleware. Роутинг на фронте уже ролевой, бэкенд-фильтрация уже работает — нужен только auth-слой.

**Q: Почему не LangChain / не векторная БД?**
> A: Over-engineering для 15 документов. Свой pipeline = 0 зависимостей, полный контроль, проще отлаживать. При росте — легко заменить.

**Q: А реально работает AI?**
> A: Да, через OpenRouter (Gemini 2.5 Flash). Без токена — fallback на контекстные mock-ответы с теми же фичами (кликабельные ссылки, markdown).
