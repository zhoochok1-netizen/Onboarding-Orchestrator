# Onboarding Orchestrator

> Современная платформа оркестрации онбординга сотрудников с AI-ассистентом, RAG-поиском по базе знаний и ролевой моделью доступа.

**Live:** [Deployed URL] &nbsp;|&nbsp; **Repo:** [github.com/zhoochok1-netizen/Onboarding-Orchestrator](https://github.com/zhoochok1-netizen/Onboarding-Orchestrator)

---

## Проблема

Онбординг затрагивает множество людей: HR оформляет документы, IT выдаёт доступы, руководитель ставит задачи, наставник проводит встречи. Без единой системы задачи теряются, сроки нарушаются, новичок не понимает, что происходит. Нужен **оркестратор**, который координирует всех участников и даёт новичку единую точку входа.

## Решение

**Onboarding Orchestrator** — full-stack веб-сервис с:
- Маршрутами адаптации по ролям (шаблоны → персональные онбординги)
- Доской задач с SLA-трекингом (зелёный/жёлтый/красный)
- AI-чатботом, знающим контекст каждого пользователя
- RAG-навигатором по базе знаний компании (15 документов, 86 чанков)
- Ролевой моделью: 5 ролей с разным интерфейсом и фильтрацией данных

### RAG Pipeline (Retrieval-Augmented Generation)

```
Запрос пользователя
       │
       ▼
┌─────────────┐    ┌──────────────┐    ┌─────────────┐
│  Tokenize   │───▶│  TF-IDF      │───▶│  Top-K      │
│  (RU-aware) │    │  Score       │    │  Chunks     │
└─────────────┘    └──────────────┘    └──────┬──────┘
                                              │
                                              ▼
                                    ┌─────────────────┐
                                    │  Context Window  │
                                    │  (≤5 chunks)     │
                                    └────────┬────────┘
                                             │
                                    ┌────────▼────────┐
                                    │  LLM Generation  │
                                    │  + Source Links   │
                                    └─────────────────┘
```

- **15 документов** → **86 чанков** (разбиение по секциям)
- **792 токена** в словаре (русскоязычный)
- TF-IDF скоринг с бустом за exact match
- Только релевантные чанки отправляются в LLM (экономия токенов)

## Функциональность

### 1. Маршрут онбординга (шаблоны)
HR создаёт шаблоны по ролям. Каждый шаблон содержит этапы (День 1 → Неделя 1 → Месяц 1) и задачи с ответственным, дедлайном и SLA. Реализовано **4 шаблона**: Разработчик, Маркетолог, Менеджер по продажам, Дизайнер.

### 2. Запуск онбординга
При добавлении сотрудника создаётся персональный маршрут из шаблона. Задачи автоматически назначаются участникам по их ролям. **10 активных онбордингов** с разной степенью прогресса.

### 3. Доска задач
Каждый участник видит свои задачи по всем новичкам. Статусы: ожидает / в работе / выполнено / просрочено. Фильтрация по ответственному и статусу. Смена статуса через выпадающий список.

### 4. SLA-трекинг
- 🟢 Зелёный — в срок
- 🟡 Жёлтый — подходит дедлайн (≤2 дня)
- 🔴 Красный — просрочено

Дашборд HR показывает сводку по всем онбордингам с SLA-барами.

### 5. AI-чатбот
Единая точка входа для новичка (и HR). Чатбот:
- Знает контекст пользователя (роль, этап, задачи, прогресс)
- Отвечает на вопросы о процессах, задачах, политиках
- Ссылается на конкретные задачи, онбординги и документы (кликабельные ссылки → модалки)
- Использует RAG для поиска по базе знаний
- Форматирует ответы в Markdown
- Сохраняет историю переписки

### 6. Навигатор по базе знаний
15 документов (СОП, ПВТР, ЛНА, гайды). RAG-поиск с TF-IDF скорингом. AI генерирует ответы с указанием источника.

## Ролевая модель

| Роль | Видит | Навигация |
|------|-------|-----------|
| **HR** | Все онбординги, все задачи, дашборд, шаблоны, чат | Полная |
| **Руководитель** | Онбординги и задачи своего отдела | Без чата |
| **IT** | Только свои задачи | Задачи + база знаний |
| **Наставник** | Задачи + подопечные | Задачи + онбординги |
| **Новичок** | Свой онбординг + свои задачи + чат | Ограниченная |

Фильтрация реализована на бэкенде через заголовок `X-User-Id`. Каждый API-эндпойнт возвращает данные, релевантные текущему пользователю.

## Принятые решения

### Почему mock-данные, а не БД?
За 1.5 дня реализовать полноценную БД с миграциями нереально без потери качества UI/AI. Mock-данные позволяют показать **все сценарии** (просрочки, прогресс, разные этапы) без инфраструктуры. Архитектура готова к замене на PostgreSQL — нужно только подменить data layer.

### Почему собственный RAG, а не LangChain?
- Полный контроль над пайплайном
- Нет тяжёлых зависимостей (LangChain = 100+ пакетов)
- TF-IDF достаточен для 15 документов
- При масштабировании можно заменить на FAISS/Chroma без изменения API

### Почему OpenRouter, а не прямой API?
OpenRouter даёт единый интерфейс к 100+ моделям. Можно переключиться с Gemini на Claude/GPT одной строкой в `.env`.

### Почему dark theme по умолчанию?
Современный тренд + удобнее для разработчиков. Есть переключатель на светлую тему.

## Запуск

### Docker (рекомендуется)

```bash
cp .env.example .env
# Отредактируйте .env — добавьте GATEWAY_TOKEN (OpenRouter)

docker-compose up --build
```

Приложение доступно на http://localhost:3000

### Локально (два терминала)

**Бэкенд:**
```bash
cd backend
pip install -r requirements.txt
cp ../.env.example ../.env  # добавьте GATEWAY_TOKEN
uvicorn main:app --reload --port 8000
```

**Фронтенд:**
```bash
cd frontend
npm install
npm start
```

Swagger-документация API: http://localhost:8000/docs

### Переменные окружения

| Переменная | По умолчанию | Описание |
|-----------|-------------|----------|
| `GATEWAY_TOKEN` | — | API-ключ OpenRouter (без него — mock-ответы чата) |
| `GATEWAY_BASE_URL` | `https://openrouter.ai/api/v1` | Базовый URL AI-провайдера |
| `GATEWAY_MODEL` | `google/gemini-2.5-flash` | Модель для генерации |
